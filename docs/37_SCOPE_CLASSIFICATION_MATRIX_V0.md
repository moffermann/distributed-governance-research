# Scope Classification Matrix v0

## Purpose

This document freezes the first scope classification matrix for the Distributed Governance System v0.

It classifies the current architecture, documents, and hypothesis clusters as:

```text
Core v0
Extension v1+
Country implementation
Out of scope
```

This matrix is not a final academic taxonomy. It is a practical scope-control tool so the project can stop expanding indefinitely and move into formal diagrams, failure-mode testing, and paper architecture.

## Classification principle

> Core v0 contains only what is necessary to make the first coherent model work from project idea to closure.

A concept belongs in Core v0 only if removing it breaks the first coherent version of the system.

## Classification categories

### Core v0

Necessary for the first coherent version of the model.

Examples:

- project object;
- project phases where needed for staged funding, deliverables, verification, and disbursement;
- actor and role model;
- proportional threshold policies;
- Planning Scope Alignment;
- Primary Responsibility Anchor;
- value thesis;
- Project Evidential Contract;
- fulfillment/control evidence needs;
- contextualized evidence items with required evidence context;
- fiscalization;
- funding commitment;
- funding attempts and bounded funding windows;
- disbursement;
- complaints;
- lifecycle states;
- related-party declaration and proportional conflict handling;
- technical audit trail.

### Extension v1+

Useful or promising, but not required for the first coherent version.

Examples:

- advanced delegation;
- public value return per peso;
- advanced benchmarking;
- complex common-good sub-charters;
- advanced capture scoring.

### Country implementation

Depends on legal, administrative, institutional, fiscal, or political decisions of a specific country.

Examples:

- tax authority integration;
- legal custody of funds;
- official identity requirements;
- pilot sector;
- administrative appeal rules;
- percentage of public budget migrated.
- budget liquidity smoothing where allowed by fiscal law or authorized implementation.

### Out of scope

Not necessary to solve in this research model now.

Examples:

- full replacement of the state;
- full constitutional design;
- national deployment across all public functions;
- complete legal code for implementation;
- universal ranking of all moral values.

## Core v0 hypothesis clusters

|---|---|---|
| Functional layer distribution principle `H001` | Core v0 framing | Needed to prevent the model from treating inherited institutions or policy sectors as indivisible blocks. H001 defines the test for deciding which layers of a function should remain centralized, become hybrid, or become distributed: public guarantees, civil rights, standards, accreditation, coercion, rights deprivation, legal recognition, macro fiscal stability, funding, service provision, execution, evidence, fiscalization, evaluation, and accountability have different risk profiles. It also distinguishes centralized legal frameworks from centralized state service provision, and platform consequences from legal consequences. |
| Foundational critique and comparative model hypotheses | Core v0 framing | Needed to explain why the model exists. |
| Bounded distributed resource allocation `H002` | Core v0 framing / financing boundary | Needed to explain that distributed allocation applies only to the configured assignable public-purpose share, inside active planning scopes, eligibility rules, threshold policies, evidence, fiscalization, phase gates, continuity-risk treatment where applicable, and audit controls. It preserves the non-assignable common pool, protected essential floors, reserve-backed or default-assigned continuity lanes where applicable, and does not treat popularity, funding, or algorithmic ordering as project legitimacy, execution readiness, disbursement approval, fulfillment proof, or indefinite service continuity. |
| Incentive-compatible distributed trust `H003` | Core v0 framing / cross-cutting principle | Needed to explain why the architecture can rely on distributed actors without collapsing into blind faith: trust comes from role-specific incentives, verifiable material claims, contextualized evidence, contradiction paths, responsible review, consequences, and auditability. |

Core v0 reads the indicators from existing OperatingMode, PlanningScope, GovernanceResolution, ReviewTimeoutResolution, and administrative observability records: scope share opened; approval, rejection, and timeout rates; rejection-reason comparability; authority-linked operator participation; public-operator privileges, subsidies, guarantees, or data access; independent versus controlled fiscalization rates; and pilot outcomes compared with the incumbent baseline. Core v0 cannot compel institutional transition or protect the migrated budget share; that political limit is recorded openly, and the fiscal channel is attacked separately in `A021`. |
| Disbursement-gaming tests `A017` | Core v0 boundary with Extension v1+ / country implementation | Needed to prevent actors from designing milestones, evidence, guarantees, or phase gates that release money before real value is delivered, without adding new payment machinery. Core v0 binds the tests through DisbursementMilestonePlan, ProjectPhase, BudgetLine, FinancialAssuranceProfile, GuaranteeMaterializationRecord, FiscalizationReport, and Disbursement: milestone linkage to budget line, phase, value relevance, evidence need, review basis, blocker check, retention, and release amount; protected advance payment; designer/modeler responsibility review for avoidably weak milestone design; fiscalizer release justification; and citizen-facing separation of funding, reservation, release, retained funds, and guarantee status. Per-project actuarial guarantee calibration and automated milestone-design scoring remain Extension v1+ / country implementation; milestone quality rests on human review backed by designer responsibility and retention. |
| Collusion observability across role clusters `A018` | Core v0 boundary with Extension v1+ / country observability | Needed to prevent formally separate roles and suppliers from coordinating to satisfy rules while defeating accountability, without promising impossible perfect detection. Core v0 reads observability from the A014 relationship-and-control graph reused across roles and suppliers, FiscalizationAssignment history, VerifiedDiscovery, ResponsibilityEvent, and AuditEvents: repeated actor-cluster visibility across projects; timing-anomaly and outcome-pattern review surfaces feeding human review; independent corroboration for critical milestones; the verified-discovery route for hidden coordination; and cross-role responsibility events where collusion is confirmed. Automated collusion detection, network-analysis scoring, and investigative capability remain Extension v1+; prosecution remains external law, and off-platform coordination among formally compliant actors can remain invisible, so the claim is comparative rather than elimination. |
| Allocation Mandate record on Planning Scope `A019` | Core v0 boundary with Extension v1+ / country implementation | Needed to prevent an unauthorized reallocation of the public purse from passing as authorized merely because it is visible and versioned, without the platform constructing the mandate itself. Core v0 records the democratic mandate on the existing Planning Scope, distinct from the procedural Administrative Rule Change machinery: mandate source (statute, ordinance, referendum, or delegated authority); the legal instrument authorizing budget migration; the migrated share; the allocation-amount formula version with an explicit flag and authority when it departs from equal-per-citizen; and effective dates with version history, with AuditEvents on every change. A non-equal formula is a higher-authorization decision than an ordinary parameter edit. Mandate construction, validation, and adjudication remain external constitutional law and country implementation, and the democratic-deficit critique of substituting atomized allocation for representative appropriation remains a recorded open normative debate. |
| Planning-scope agenda observability `A020` | Core v0 boundary with Extension v1+ / open question | Needed to prevent agenda-setting capture — whoever constructs Planning Scopes decides what citizens may fund at all before allocation, support, or fiscalization occurs — without pretending Core v0 solves roadmap-construction governance. Core v0 adds no scope-construction entity; it carries interim observability over existing objects: every Planning Scope shows its Allocation Mandate record (A019) and a versioned scope definition; scope changes and interpretations surface as Governance Resolutions with declared review windows and timeout policies (C020); and out-of-scope demand stays visible as Ideas (C001) with volume observability per rejected or out-of-scope category. Agenda-setting power is documented, versioned, and contestable through visibility only; distributing who constructs the roadmap remains the acknowledged constitutional gap and a blocker for open-mode deployment at scale. |
| Fiscal Commitment Profile and fiscal-reliability observability `A021` | Core v0 boundary with Extension v1+ / country implementation | Needed to prevent the state from strangling the system through the fiscal channel it still controls — late or partial delivery of signed balances, mid-cycle reduction of the migrated share, zero indexation, or unexecuted valid orders — while the platform shows green funding states, without adding any enforcement mechanism software cannot provide. Core v0 adds no fiscal-enforcement entity; it records a public, versioned Fiscal Commitment Profile on the existing Planning Scope (migrated percentage, indexation rule, delivery-latency target, cycle horizon) whose changes are governance events with reason, magnitude, affected scopes, and beneficiary impact, and surfaces reliability indicators from custody and Financial Order records — expected-versus-actual delivery, order-to-execution latency, an unexecuted-valid-order indicator, and a real-value-preservation warning — feeding the H054 transition metrics and A016 incumbent-resistance indicators. Fiscal strangulation becomes measurable and attributable to the fiscal authority rather than the visible executor; enforcement stays external, and a determined treasury can still defund the system in full public view. |
| Duty-of-Care Anchor and third-party coverage `A033` | Core v0 boundary with Extension v1+ / country implementation | Needed to prevent a responsibility vacuum where harm to a third party results from the interaction of adequately-performed roles — the responsibility matrix truthfully finds no internal breach, C012 forbids blame by association, and P003 assigns responsibility to "the project," which is not a legal person a victim can sue — without the platform adjudicating liability. Core v0 adds no liability entity; every execution-financeable project names, before disbursement, the solvent, reachable legal person or persons answerable to third parties for physical and safety harm (executor, guarantor, insurer, or designated responsible operator), exposed on the citizen-facing sheet; the Financial Assurance and Post-Closure Coverage profiles extend to third-party liability coverage proportionate to physical risk where Threshold Policy requires; versioned documentation records that distributed micro-allocation does not dilute the state's duty of care; the responsibility matrix distinguishes "no role breached its internal obligation" from "no one is liable to the victim"; and the audit trail preserves a court-usable harm-attribution package. A victim starts from a named, solvent defendant rather than a doctrinal vacuum; liability apportionment stays external law, and a named anchor guarantees a reachable defendant, not a just outcome. |
| Control-supply density observability `A022` | Core v0 boundary with Extension v1+ / country implementation | Needed to prevent control assumptions from silently failing where no independent, qualified fiscalizer or evidence producer exists at a price the project can bear, without the platform manufacturing supply. Core v0 adds no control-supply entity; supply-density signals attach to PlanningScope, Control Subproject selection, and FiscalizationAssignment records — eligible fiscalizers and evidence producers per scope, offer rates, fee levels, and repeat-assignment concentration — and a thin-market indicator names a structural gap before publication rather than at control-package closure. Thin-market handling is a disclosed country-implementation choice — remote and documentary evidence, cross-territory and pooled assignment, and relaxed-but-disclosed selection under stronger safeguards — with priced compensation for weak verification raising the guarantee rather than lowering the standard. ## Classification rules for future additions

When a new idea appears, classify it using this test:

```text
1. Does v0 fail without it?
   Yes → Core v0 candidate.
   No → continue.

2. Is it useful but adds complexity?
   Yes → Extension v1+.
   No → continue.

3. Does it depend on law, state design, budget, or jurisdiction?
   Yes → Country implementation.
   No → continue.

4. Is it unnecessary for the model?
   Yes → Out of scope.
```

## Core v0 closure test

Core v0 is considered bounded if it can explain:

```text
actor
→ role
→ project
→ value promise
→ funding commitment
→ Project Evidential Contract
→ fulfillment/control evidence need
→ contextualized evidence item with required evidence context
→ fiscalization / formal evaluation
→ milestone
→ disbursement
→ complaint/correction if needed
→ closure
→ reputation
→ audit trail
```

If a concept does not materially improve this chain, it should not enter Core v0.
