# Scope Classification Matrix v0

## Purpose

This document freezes the first scope classification matrix for the Distributed Governance System v0.

It classifies the current architecture, documents, and hypothesis clusters as:

```text
Core v0
Extension v1+
Country implementation
Out of scope
```

This matrix is not a final academic taxonomy. It is a practical scope-control tool so the project can stop expanding indefinitely and move into formal diagrams, failure-mode testing, and paper architecture.

## Classification principle

> Core v0 contains only what is necessary to make the first coherent model work from project idea to closure.

A concept belongs in Core v0 only if removing it breaks the first coherent version of the system.

## Classification categories

### Core v0

Necessary for the first coherent version of the model.

Examples:

- project object;
- project phases where needed for staged funding, deliverables, verification, and disbursement;
- actor and role model;
- proportional threshold policies;
- Planning Scope Alignment;
- Primary Responsibility Anchor;
- value thesis;
- Project Evidential Contract;
- fulfillment/control evidence needs;
- contextualized evidence items with required evidence context;
- fiscalization;
- funding commitment;
- funding attempts and bounded funding windows;
- disbursement;
- complaints;
- lifecycle states;
- related-party declaration and proportional conflict handling;
- technical audit trail.

### Extension v1+

Useful or promising, but not required for the first coherent version.

Examples:

- advanced delegation;
- public value return per peso;
- advanced benchmarking;
- complex common-good sub-charters;
- advanced capture scoring.

### Country implementation

Depends on legal, administrative, institutional, fiscal, or political decisions of a specific country.

Examples:

- tax authority integration;
- legal custody of funds;
- official identity requirements;
- pilot sector;
- administrative appeal rules;
- percentage of public budget migrated.
- budget liquidity smoothing where allowed by fiscal law or authorized implementation.

### Out of scope

Not necessary to solve in this research model now.

Examples:

- full replacement of the state;
- full constitutional design;
- national deployment across all public functions;
- complete legal code for implementation;
- universal ranking of all moral values.

## Core v0 hypothesis clusters

| Hypothesis or cluster | Classification | Reason |
|---|---|---|
| Functional layer distribution principle `H001` | Core v0 framing | Needed to prevent the model from treating inherited institutions or policy sectors as indivisible blocks. H001 defines the test for deciding which layers of a function should remain centralized, become hybrid, or become distributed: public guarantees, civil rights, standards, accreditation, coercion, rights deprivation, legal recognition, macro fiscal stability, funding, service provision, execution, evidence, fiscalization, evaluation, and accountability have different risk profiles. It also distinguishes centralized legal frameworks from centralized state service provision, and platform consequences from legal consequences. |
| Foundational critique and comparative model hypotheses | Core v0 framing | Needed to explain why the model exists. |
| Bounded distributed resource allocation `H002` | Core v0 framing / financing boundary | Needed to explain that distributed allocation applies only to the configured assignable public-purpose share, inside active planning scopes, eligibility rules, threshold policies, evidence, fiscalization, phase gates, continuity-risk treatment where applicable, and audit controls. It preserves the non-assignable common pool, protected essential floors, reserve-backed or default-assigned continuity lanes where applicable, and does not treat popularity, funding, or algorithmic ordering as project legitimacy, execution readiness, disbursement approval, fulfillment proof, or indefinite service continuity. |
| Incentive-compatible distributed trust `H003` | Core v0 framing / cross-cutting principle | Needed to explain why the architecture can rely on distributed actors without collapsing into blind faith: trust comes from role-specific incentives, verifiable material claims, contextualized evidence, contradiction paths, responsible review, consequences, and auditability. |
| Project-based governance hypotheses | Core v0 | Project is the core operating unit. |
| Personal AI guides `H005` | Extension v1+ with Core v0 compatibility boundary | Useful for reducing participation cost, but not required for the first coherent model. Core v0 only requires that any connected guide remain assistance infrastructure, not a civic actor, delegate, automatic allocation profile, hidden recommender, or source of material action without explicit citizen confirmation or a separately auditable platform rule. |
| Assisted deliberation context `H006` | Core v0 | Needed to improve decision quality before material citizen actions while preserving citizen judgment, source visibility, correction, dissent, and the C008 rule that AI assists but does not decide. |
| Continuous performance visibility `H007` | Core v0 | Needed to make reviewed historical role performance visible before future funding, support, delegation, selection, or review decisions, while avoiding a universal score or an unbounded history for every value declared by past projects. |
| Distributed accountability loop `H008` | Core v0 | Needed to close the full project cycle from accepted promise to evidence, formal review, Project Closure Accountability Record, financial closure, role-specific responsibility, reputation, and future financing context. |
| Planning scope alignment `H009` | Core v0 with Extension/Country boundary | Needed so financeable projects align with an active public-function, pilot, operating-mode, protocol, roadmap, or authorized planning line. Core v0 does not define full distributed roadmap construction, but active scopes should expose essential-service protection and planning-continuity targets where applicable. |
| Continuity risk classification `A006` | Core v0 project/funding boundary | Needed so recurring, continuity-critical, emergency, and maintenance-dependent projects expose funded period, future funding dependency, renewal trigger, beneficiary protection, wind-down rule, and continuity-need Idea path where applicable. It does not automatically renew the current executor and remains separate from A005 Planning Scope essential-service protection. |

| Distributed value and antivalue management `H012` | Core v0 | Needed to require projects to declare value floors, antivalue ceilings, metrics, fulfillment/control evidence, affected scope, and role-specific consequences without turning antivalue review into automatic complaints or a centralized net-value score. |

## Classification rules for future additions

When a new idea appears, classify it using this test:

```text
1. Does v0 fail without it?
   Yes → Core v0 candidate.
   No → continue.

2. Is it useful but adds complexity?
   Yes → Extension v1+.
   No → continue.

3. Does it depend on law, state design, budget, or jurisdiction?
   Yes → Country implementation.
   No → continue.

4. Is it unnecessary for the model?
   Yes → Out of scope.
```

## Core v0 closure test

Core v0 is considered bounded if it can explain:

```text
actor
→ role
→ project
→ value promise
→ funding commitment
→ Project Evidential Contract
→ fulfillment/control evidence need
→ contextualized evidence item with required evidence context
→ fiscalization / formal evaluation
→ milestone
→ disbursement
→ complaint/correction if needed
→ closure
→ reputation
→ audit trail
```

If a concept does not materially improve this chain, it should not enter Core v0.
