# Core v0 Scope

## Purpose

This document freezes the first coherent scope of the distributed governance model. Its purpose is to prevent unlimited conceptual expansion and to distinguish the core architecture from future extensions and country-specific implementation choices.

## Scope classification rule

Every new idea should be classified as one of four types:

```text
A. Core v0
   Necessary for the first coherent version of the model.

B. Extension v1+
   Interesting and potentially useful, but not necessary for the first paper or core model.

C. Country implementation
   Depends on the state, jurisdiction, law, public administration, or political decision of the implementer.

D. Out of scope
   Not necessary for the model and should not be solved now.
```

## Core v0 — Minimum architecture

These modules are necessary for the first coherent version of the model.

- project as the base unit;
- project value thesis and KPIs;
- separated roles: modeler/designer, executor, fiscalizer;
- distributed financing;
- civic wallet / assignable public-purpose budget;
- public allocation amount rules and external provider boundary;
- support versus financing signals;
- funding closure once target is reached;
- milestones, guarantees, and disbursement rules;
- evidence and fiscalization;
- complaints and fiscalization triggers;
- role-based reputation;
- verified identity and contextual privacy;
- basic delegation;
- allocation profiles and automatic monthly allocation;
- transparent project discovery;
- explainable discovery ordering and user-customizable Home categories;
- minimum meta-governance and protocol evolution discipline for visible rule changes;
- progressive institutional transition;
- operating modes: closed, tutored, semi-open, open, suspended;
- simple user-facing signals and a minimal administrative observability baseline.

## Complementary v0 modules

These modules belong in the first version as complementary capabilities, but they should not be the center of the first explanation.

- composite programs as project aggregators;
- common-good impact declarations through project fields;
- urgency/emergency as a project attribute;
- tutored governance transparency during transition.

They help the architecture handle more complex cases, but the core model should be explainable without starting from them.

## Extension v1+

These concepts are documented but should not be treated as necessary for Core v0.

- subordinate common-good charters;
- full common-good governance programs;
- common-good charters and charter voting;
- subdelegation;
- explicit time-bound delegation;
- temporary delegation pause;
- third-party shareable allocation profiles;
- complex canary release mechanisms;
- advanced recommendation algorithms;
- full universal institutional observability panel;
- multiple incompatible institutional architectures;
- fully automated scope moderation;
- highly granular operating modes.

## Country implementation

These questions belong to the implementing country or jurisdiction, not the core model.

- which public function starts first;
- what percentage of budget enters the pilot;
- what happens to public employees;
- when to increase or reduce distributed budget share;
- what laws must be changed;
- which institution operates or hosts the platform;
- whether a country deliberately departs from Core v0 by allowing public institutions to act inside the platform;
- what external datasets are used for comparison;
- what political adoption pace is chosen;
- what sector-specific KPIs are used;
- what legal constraints apply to each function.

## Core explanatory blocks

The Core v0 architecture should be explained in seven blocks.

### Block 1 — Foundation

Problem, motivation, functional distribution, and why the model is not simply “more state” or “less state.”

### Block 2 — Operational unit

Project, value thesis, KPIs, evidence, roles, and responsibilities.

### Block 3 — Financing

Civic wallet, citizen amount formulas, external authorized amount providers where needed, monthly allocation, profiles, funding closure, milestones, guarantees, and disbursement.

### Block 4 — Control

Fiscalization, complaints, reputation, antivalue, sanctions, and accountability.

### Block 5 — Participation

Identity, contextual privacy, delegation, delegate reporting, and revocation.

### Block 6 — Evolution

Meta-governance, protocol change, testing environments, approval rules, and versioning.

### Block 7 — Transition

Operating modes, tutored pilots, governance resolutions, review timeout resolutions, transition metrics, and the administrative observability baseline.

## First thesis statement

A candidate thesis for the first version:

> A state can progressively distribute selected public functions through a project-based architecture that combines assignable civic budgets, explicit value theses, distributed fiscalization, role-based reputation, delegation, protocol evolution, and observable institutional transition.

## Principle

> If a concept is not necessary to explain the first coherent version of the model, it should be classified as an extension, implementation decision, or out-of-scope item.

## Status

Accepted as the working Core v0 scope boundary.
