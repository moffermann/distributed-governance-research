# H021 — Project Variation Control

## Hypothesis

An executor may introduce variations to a project design only through a transparent and proportionate change-control process. The executor remains accountable to the base design it accepted unless the variation is formally approved.

## Rationale

A project is funded based on a declared design, value thesis, KPIs, risks, budget, and expected results. If an executor can alter the project unilaterally, the original social approval loses meaning.

## Base rule

The executor responds to the KPIs and commitments defined in the base design it accepted.

## Variations

A variation may be:

- minor and merely recorded;
- relevant and requiring approval;
- substantial enough to become a new version of the project.

## Examples

### Minor variation

Small operational adjustment that does not alter cost, KPIs, value thesis, beneficiaries, risk, or antivalue.

### Relevant variation

Change that affects execution method, timeline, budget, mitigation, or secondary goals.

### Substantial variation

Change that alters the primary goal, core KPIs, declared value, affected groups, or expected antivalue. This should be treated as a new project version.

## Evaluation rule

Fiscalizers evaluate against the accepted base design and any formally approved variations.

Unapproved changes do not erase the original responsibility.

## Principle

> A project is not approved as a general intention. It is approved as a specific version with defined commitments.

## Status

Project execution control hypothesis. Inspired by current contract and public procurement logic, where major changes generally require formal approval.
