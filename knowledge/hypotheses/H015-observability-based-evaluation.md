# H015 — Observability-Based Evaluation

## Hypothesis

Only actors with a verifiable relationship to a project dimension should be able to formally evaluate that dimension.

## Rationale

A distributed system should not become an open popularity poll where anyone can rate anything. If a person cannot observe, measure, or verify a dimension, then their rating would be speculative.

Citizens may always express opinions, alerts, support, or criticism. But formal evaluation should be tied to observability.

## Principle of observability

> If an actor cannot observe, measure, or verify a dimension, that actor should not formally score that dimension.

## Role examples

### Beneficiaries

Can evaluate:

- received value;
- treatment;
- usefulness;
- perceived outcome;
- service experience.

### Affected parties

Can evaluate:

- externalities;
- quality-of-life impacts;
- harms;
- local effects;
- unfulfilled mitigation.

### Financial auditors

Can evaluate:

- use of resources;
- traceability;
- deviations;
- financial consistency;
- misuse of funds.

### Technical fiscalizers

Can evaluate:

- technical compliance;
- environmental performance;
- safety;
- engineering standards;
- sanitary standards;
- project-specific KPIs.

### Experts

Can evaluate:

- methodology;
- technical plausibility;
- design quality;
- risks;
- coherence with evidence.

### General citizens

Can:

- comment;
- support;
- finance;
- raise alerts;
- request fiscalization;
- affect public perception;
- but should not formally score dimensions they cannot observe.

## Implication

The system should distinguish between:

```text
public opinion
beneficiary evaluation
affected-party evaluation
technical fiscalization
financial audit
expert review
```

## Status

Core evaluation design principle.
