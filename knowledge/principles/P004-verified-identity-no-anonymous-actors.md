# P004 — Verified Identity and No Anonymous Actors

## Principle

A distributed governance system should not allow anonymous actors in formal participation, funding, project execution, fiscalization, complaint, reputation, or protocol-change processes.

## C024 alignment

C024 preserves this principle and clarifies how sensitive participation should work.

Formal alerts and complaints should not be anonymous. When public exposure would create retaliation, safety, privacy, or beneficiary-protection risks, the actor may request protected identity. In that case:

- the actor remains verified to the system;
- the public may see a contextual protected display identity;
- authorized reviewers, fiscalizers, or auditors may access restricted identity only under role, purpose, and access-log rules;
- protected identity does not create anonymous formal power.

## Rationale

If the system is intended to function as a real social contract, every actor who exercises formal power must be identifiable and accountable.

Anonymity may be compatible with discussion in ordinary online spaces, but not with a system that allocates resources, affects reputations, pauses projects, triggers fiscalization, approves protocol changes, or revokes social support.

## Formal participation requires identity

All formal actors should have verified identity:

- citizens;
- funders;
- project modelers;
- executors;
- fiscalizers;
- auditors;
- beneficiaries where applicable;
- affected parties;
- reviewers;
- protocol-change proposers;
- complaint initiators;
- voters.

## Identity mechanism example

In Chile, an identity mechanism such as ClaveUnica could serve as a conceptual reference for verified access.

## Important distinction

Verified identity does not necessarily mean all personal data must be publicly exposed. The system may separate:

- identity verification;
- public display name;
- privacy protection;
- legal/accountability traceability.

## Principle formulation

> Formal power in the system requires verified identity. No anonymous actor should be able to allocate resources, damage reputation, trigger enforcement, or modify governance rules without being accountable.

## Protected alert example

A parent in a child sports project reports that beneficiaries may face retaliation if families complain.

The parent should not file an anonymous formal alert. The parent should file as a verified actor and request protected identity. The public may see the alert as coming from a contextual protected complainant, while authorized reviewers can verify the identity under restricted access.

## Open questions

- Which actions require public identity versus private verified identity?
- How is privacy protected for vulnerable beneficiaries or whistleblowers?
- How should protected-identity alerts and complaints be audited without exposing the actor publicly?
- How is identity portability handled across jurisdictions?

## Status

Core constitutional principle of the system.
